Base de la pyramide
Tests Unitaires
Périmètre : une seule fonction / méthode / classe — en isolation totale
Sur quoi on teste ?
  • Une fonction pure (entrée → sortie)
  • Une méthode d'une classe
  • Un helper, un utilitaire, un validator
  • Un reducer (Redux, Zustand…)
  • Un composant UI isolé (React, Vue…)
Structure AAA
Arrange · Act · Assert
  • Arrange — préparer les données d'entrée
  • Act — appeler la fonction testée
  • Assert — vérifier le résultat obtenu
Chaque test = un seul comportement vérifié
Mocks & Stubs
Les dépendances externes sont remplacées pour isoler l'unité :
  • Mock : remplace une fonction et vérifie les appels
  • Stub : retourne une valeur fixe à la place d'une vraie dépendance
  • Spy : observe une vraie fonction sans la remplacer
  • Exemples : BDD, API, système de fichiers, date/heure
Data de test
On construit des données contrôlées :
  • Valeurs nominales (cas standard)
  • Valeurs limites (0, -1, null, "", [], undefined)
  • Cas d'erreur attendus (throw, reject)
  • Fixtures statiques (objets JSON en dur)
Jest / Vitest — exemple test unitaire
// ✅ Fonction à tester
function calculerTaxe(prix, taux) {
  if (prix < 0) throw new Error('Prix invalide')
  return +(prix * (1 + taux)).toFixed(2)
}

// 🧪 Le test
describe('calculerTaxe', () => {

  it('calcule le prix TTC correctement', () => {
    // Arrange
    const prix = 100, taux = 0.2

    // Act
    const result = calculerTaxe(prix, taux)

    // Assert
    expect(result).toBe(120)
  })

  it('lève une erreur si le prix est négatif', () => {
    expect(() => calculerTaxe(-10, 0.2)).toThrow('Prix invalide')
  })
})

Milieu de la pyramide
Tests d'Intégration
Périmètre : plusieurs modules qui collaborent — API, BDD, services
Sur quoi on teste ?
  • Route API + contrôleur + service
  • Service + accès BDD (requêtes SQL/ORM)
  • Module A qui appelle le module B
  • Queue de messages (pub/sub)
  • Composants UI + store + API mock
Qu'est-ce qu'on remplace ?
  • On garde les modules à tester ensemble
  • On mock ce qui est hors périmètre (services tiers, emails…)
  • BDD : en mémoire (SQLite) ou conteneur Docker
  • HTTP externe : intercepté avec MSW ou nock
Data de test
  • Fixtures insertées avant le test (seed)
  • BDD nettoyée entre chaque test
  • Factories pour générer des objets cohérents
  • Payloads JSON réalistes (proche de la prod)
Jest + Supertest — test d'une route API
import request from 'supertest'
import app from '../app'
import { db } from '../db'

beforeEach(async () => {
  await db.seed()              // injecter des données de test
})

afterEach(async () => {
  await db.clean()             // remettre à zéro
})

it('GET /users/:id → retourne l\'utilisateur', async () => {
  const res = await request(app)
    .get('/users/42')
    .set('Authorization', 'Bearer test-token')

  expect(res.status).toBe(200)
  expect(res.body).toMatchObject({
    id: 42,
    email: 'alice@test.com'
  })
})

Haut de la pyramide
Tests Fonctionnels
Périmètre : fonctionnalité métier complète — du point de vue utilisateur
Sur quoi on teste ?
  • Un scénario métier de bout en bout (ex: passer une commande)
  • Le comportement d'une feature complète
  • Les règles de gestion (ex: promo si panier > 50€)
  • Les cas nominaux et les cas d'erreur métier
  • Peut être UI ou purement API
Différence avec l'intégration
Les tests d'intégration vérifient que les modules communiquent. Les tests fonctionnels vérifient que la feature répond aux specs métier. On pense en termes de :
  • "En tant qu'utilisateur, quand je... alors..."
  • User stories / critères d'acceptance
  • BDD : Given / When / Then (Cucumber)
Mocks & environnement
  • API tierces mockées (paiement, email, SMS)
  • BDD de test avec données représentatives
  • Auth simulée avec tokens de test
  • Date/heure parfois fixée (sinon tests instables)
Data & assertions
  • Scénarios complets avec données réalistes
  • Assertions sur le résultat métier, pas technique
  • Ex: "la commande est bien en état CONFIRMED"
  • Ex: "l'email de confirmation a été envoyé (mock)"
Playwright — test fonctionnel d'ajout au panier
// Scénario métier : "Un utilisateur connecté peut ajouter un produit"
test('ajout au panier → badge mis à jour', async ({ page }) => {

  // Arrange — contexte utilisateur connecté
  await page.goto('/produits/sneaker-42')

  // Act — interaction utilisateur
  await page.click('[data-testid="btn-ajouter"]')

  // Assert — résultat métier visible
  await expect(page.locator('.cart-badge'))
    .toHaveText('1')

  await expect(page.locator('.toast'))
    .toContainText('Ajouté au panier')
})

Sommet de la pyramide
Tests End-to-End (E2E)
Périmètre : navigateur → frontend → backend → BDD — aucun mock
Sur quoi on teste ?
  • Parcours utilisateur complets et critiques
  • Ex: inscription → login → achat → confirmation email
  • Interaction réelle avec un navigateur (Chromium, Firefox)
  • Stack complète sans aucun remplacement
  • Environnement de staging (proche de la prod)
Aucun mock — tout est réel
C'est la force et la faiblesse des tests E2E :
  • ✅ Valide l'intégration complète du système
  • ✅ Teste ce que l'utilisateur vit vraiment
  • ⚠️ Lents (secondes par test)
  • ⚠️ Fragiles (dépendent du réseau, UI, timing)
  • ⚠️ Coûteux à maintenir
Playwright — parcours complet login + achat
test('parcours complet : login → achat → confirmation', async ({ page }) => {

  // 1. Login réel
  await page.goto('/login')
  await page.fill('[name=email]', 'alice@test.com')
  await page.fill('[name=password]', 'secret')
  await page.click('button[type=submit]')
  await page.waitForURL('/dashboard')

  // 2. Ajout produit + commande
  await page.goto('/produits/42')
  await page.click('text=Ajouter au panier')
  await page.click('text=Commander')

  // 3. Vérification confirmation
  await expect(page).toHaveURL(/\/confirmation/)
  await expect(page.locator('h1')).toContainText('Commande confirmée')
})

Test spécialisé
Tests Smoke 🔥
Périmètre : fonctions vitales uniquement — go / no-go post-déploiement
Qu'est-ce qu'on vérifie ?
  • L'application démarre et répond
  • Les endpoints critiques (health check, login…)
  • La page d'accueil se charge sans erreur 500
  • Les connexions BDD / cache / services essentiels
  • Rien de plus — c'est volontairement minimal
Comment ça s'exécute ?
  • Sous-ensemble tagué de tests existants
  • Lancé automatiquement après chaque déploiement CI/CD
  • Doit passer en < 2-3 minutes
  • Si ça échoue → rollback immédiat
  • Résultat binaire : 🟢 OK ou 🔴 KO
Playwright — suite smoke taggée
// playwright.config.ts → tag @smoke pour filtrer
test('@smoke — health check API', async ({ request }) => {
  const res = await request.get('/api/health')
  expect(res.status()).toBe(200)
})

test('@smoke — page d\'accueil accessible', async ({ page }) => {
  await page.goto('/')
  await expect(page).not.toHaveTitle(/Error/)
})

test('@smoke — login fonctionnel', async ({ page }) => {
  await page.goto('/login')
  await expect(page.locator('form')).toBeVisible()
})

// Exécution : npx playwright test --grep @smoke

Test spécialisé — Performance
Tests de Charge / Stress ⚡
Périmètre : infrastructure · API · comportement sous pression
Test de charge (Load)
Simule un nombre normal à élevé d'utilisateurs simultanés pour mesurer les performances en conditions réelles.
  • Mesure : temps de réponse, débit (req/s)
  • Montée progressive en charge
  • Vérifie le respect des SLAs
Test de stress
Pousse le système au-delà de ses limites pour trouver le point de rupture et observer comment il se comporte.
  • Charge x5, x10 le nominal
  • Détecte fuites mémoire, deadlocks
  • Vérifie la récupération après pic
Test d'endurance (Soak)
Charge soutenue dans le temps (heures / jours) pour détecter les dégradations progressives.
  • Fuites mémoire qui s'accumulent
  • Connexions BDD qui s'épuisent
  • Disque qui se remplit
k6 — scénario de charge sur une API
import http from 'k6/http'
import { check, sleep } from 'k6'

export const options = {
  stages: [
    { duration: '30s', target: 50  },  // montée à 50 utilisateurs
    { duration: '1m',  target: 200 },  // maintien à 200
    { duration: '20s', target: 0   },  // descente
  ],
  thresholds: {
    'http_req_duration': ['p(95)<500'], // 95% des req < 500ms
    'http_req_failed':   ['rate<0.01'], // moins de 1% d'erreurs
  }
}

export default function() {
  const res = http.get('https://api.monapp.com/products')

  check(res, {
    'status 200': (r) => r.status === 200,
    'body non vide': (r) => r.body.length > 0,
  })

  sleep(1)
}
Métriques clés
  • p95 / p99 — percentile des temps de réponse
  • RPS — requêtes par seconde
  • Error rate — taux d'erreurs
  • TTFB — time to first byte
  • CPU, RAM, connexions BDD (monitoring)
Environnement requis
  • Environnement dédié, jamais la prod
  • Infrastructure identique à la prod
  • Monitoring activé (Grafana, Datadog…)
  • BDD pré-peuplée de données réalistes
  • Isolation réseau (ne pas tester des tiers)

Stratégie transversale
Tests de Non-Régression (TNR) 🔄
Pas un type de test : une stratégie d'exécution systématique après chaque modification
Le principe
Après chaque modification du code (feature, bugfix, refacto), on rejoue une suite de tests pour s'assurer que rien de ce qui fonctionnait avant n'est cassé. La TNR n'a pas son propre cadre technique : elle orchestrer les tests existants.
Quels tests inclure ?
Le périmètre dépend du risque de la modification :
  • Petite correction → tests unitaires + intégration
  • Nouvelle feature → ajout des tests fonctionnels
  • Refacto majeur → suite complète + E2E
  • Déploiement → smoke test en premier
GitHub Actions — pipeline TNR automatisé
# .github/workflows/ci.yml
on: [push, pull_request]

jobs:
  regression:
    steps:
      - name: Tests unitaires
        run: npx vitest run

      - name: Tests d'intégration
        run: npx jest --testPathPattern=integration

      - name: Tests fonctionnels E2E
        run: npx playwright test

      - name: Smoke post-deploy      # déclenché après merge sur main
        run: npx playwright test --grep @smoke
        if: github.ref == 'refs/heads/main'
Bonnes pratiques
  • Automatiser dans la CI — jamais manuel
  • Les tests doivent être déterministes (pas de flaky tests)
  • Versioner les tests avec le code
  • Toute PR doit faire passer la TNR
  • Surveiller la durée : si trop long → paralléliser
TNR ≠ Recette (UAT)
La TNR est automatique et technique. Elle garantit la stabilité interne. La recette (User Acceptance Testing) est humaine et métier : le client/PO valide que la feature répond au besoin. Les deux sont complémentaires.
